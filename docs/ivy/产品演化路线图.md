## 1. 战略概述

### 1.1 核心理念

**从"集体养育"到"个性陪伴"的演化路径**

- 🌱 **起点**：所有用户共同养育一个小糖（社区感）
- 🌿 **演化**：分组社区，平衡社区感和亲密感
- 🌳 **成熟**：免费社区 + 付费专属，商业闭环
- 🌲 **深化**：社交网络，内容生态

### 1.2 设计哲学

| 原则 | 说明 |
|------|------|
| **渐进式演化** | 技术架构支持平滑过渡，不推倒重来 |
| **价值验证优先** | 先验证核心价值，再扩展功能 |
| **商业化准备** | 从MVP开始就为商业化预留空间 |
| **用户体验第一** | 功能演化服务于用户需求，不为技术而技术 |

## 3. Phase 2: 分组社区模式

> ⚠️ **初稿状态**：本章节为初步规划，未经详细讨论和验证，待Phase 1完成后重新评估。

### 3.1 时间线

**目标周期**: 3-6个月  
**触发条件**: Phase 1成功指标达成

### 3.2 产品形态

**核心概念**：用户被分配到不同社区，每个社区共同养育自己的小糖

```
┌───────────────┐  ┌───────────────┐  ┌───────────────┐
│ 社区A (100人) │  │ 社区B (100人) │  │ 社区C (100人) │
│               │  │               │  │               │
│  🐱 小糖A     │  │  🐱 小糖B     │  │  🐱 小糖C     │
│  能量: 80     │  │  能量: 60     │  │  能量: 90     │
│  饥饿: 30     │  │  饥饿: 70     │  │  饥饿: 20     │
└───────────────┘  └───────────────┘  └───────────────┘
        ↓                  ↓                  ↓
   用户1-100          用户101-200        用户201-300
```

### 3.3 分组策略

#### 策略A: 按注册时间分组（推荐）
- 每100个用户自动分配到一个新社区
- 优势：实现简单，用户体验平滑
- 劣势：无法选择社区

#### 策略B: 按地理位置分组
- 同城/同省用户在同一社区
- 优势：社区归属感强
- 劣势：地域分布不均，某些社区人数过多/过少

#### 策略C: 按活跃时段分组
- 夜猫子社区（22:00-6:00活跃）
- 早起社区（6:00-22:00活跃）
- 优势：用户互动时间重叠
- 劣势：用户作息改变怎么办？

**MVP建议**: 先采用策略A，简单有效

### 3.4 核心功能

| 功能 | 描述 | 优先级 |
|------|------|--------|
| **社区分配** | 新用户自动分配到社区 | P0 |
| **社区排行榜** | 社区活跃度排行 | P1 |
| **社区聊天** | 社区成员简单交流（可选） | P2 |
| **社区切换** | 用户申请更换社区（限制频率） | P2 |

### 3.5 技术架构

```python
# Redis Key设计
life:community-{id}:rhythm      # 社区宠物状态
life:community-{id}:energy

# 用户-社区关系
user:{user_id}:community_id          # 用户属于哪个社区
community:{id}:members (Set)         # 社区成员列表
community:{id}:push_tokens (Set)     # 社区推送token列表

# 社区元数据
community:{id}:created_at
community:{id}:total_interactions    # 社区总互动次数
```

```python
# 定时任务
@cron("* * * * *")  # 每分钟
def tick_all_communities():
    """推进所有社区宠物的生命"""
    community_ids = get_active_communities()  # 假设100个
    
    for community_id in community_ids:
        life = Life(backend='redis', instance_id=f'community-{community_id}')
        life.start()
        life.tick()
        life.flush()
        life.stop()
        
        # 判断是否触发主动行为
        if should_trigger_active_behavior(life):
            send_push_to_community(community_id)
```

### 3.6 资源消耗

| 指标 | 数值 | 说明 |
|------|------|------|
| **Life实例数** | 100-500个 | 取决于社区数 |
| **定时任务执行时间** | <10秒 | 100个社区 × 10ms |
| **Redis存储** | <500KB | 100个社区 × 5KB |
| **推送成本** | O(社区数) × O(社区人数) | 分组推送，效率更高 |

### 3.7 用户体验变化

**优势**：
- ✅ 社区归属感更强（100人 vs 10000人）
- ✅ 互动更有存在感（我的喂食更"有用"）
- ✅ 推送更精准（社区A饿了，只通知社区A）

**挑战**：
- ⚠️ 新用户体验差异（早期社区 vs 新建社区）
- ⚠️ 社区活跃度不均（有的社区很活跃，有的冷清）
- ⚠️ 社区切换需求（用户想换到朋友的社区）

### 3.8 触发下一阶段的条件

**满足以下任一条件**：
1. ✅ 付费意愿调研结果良好（>30%用户愿意付费）
2. ✅ DAU > 5000，月活 > 20000
3. ✅ 运营成本压力（需要商业化）
4. ✅ 完成6个月运营周期

### 4.3 功能对比

| 功能 | 免费（社区宠物） | 付费（专属宠物） |
|------|----------------|----------------|

| **推送** | 社区推送 | 专属推送 + 社区推送 |

### 4.4 定价策略

#### 方案A: 订阅制（推荐）

| 套餐 | 价格 | 功能 | 目标用户 |
|------|------|------|----------|
| **免费版** | ¥0 | 社区宠物 | 所有用户 |
| **基础版** | ¥6/月 | 专属宠物 | 轻度付费用户 |
| **进阶版** | ¥18/月 | 专属宠物 + 装扮包 | 中度付费用户 |
| **年费版** | ¥68/年 | 全功能（相当于¥5.67/月） | 重度用户 |

#### 方案B: 一次性购买 + IAP

- 专属宠物解锁：¥18（一次性）
- 装扮包：¥6-12/个（IAP）
- 房间装饰：¥6-18/套（IAP）

**推荐**: 方案A（订阅制），理由：
- 持续现金流
- 用户心理负担小（¥6/月 vs ¥18一次性）
- 便于运营（会员专属活动）

### 6.3 micro-life-sim的稳定性

**关键设计决策**：
```python
# micro-life-sim API（不变）
Life(backend='redis', instance_id='xxx')
```

**instance_id的抽象性**：
- Phase 1: `instance_id='global'`
- Phase 2: `instance_id='community-1'`

## 10. 附录

### A. 参考案例

#### A.1 成功案例
- **Tamagotchi**：电子宠物鼻祖，验证了宠物养成的市场需求
- **旅行青蛙**：佛系养成，证明轻度互动也有市场
- **Walkr**：计步器+宠物，证明功能性和趣味性可以结合

#### A.2 失败案例
- **过度复杂的养成系统**：用户学习成本高，流失率高
- **过度依赖推送**：打扰用户，导致卸载
- **缺乏长期目标**：新鲜感过后无事可做

### B. 数据指标定义

| 指标 | 定义 | 计算公式 |
|------|------|----------|
| **DAU** | 日活跃用户数 | 当日打开App的独立用户数 |
| **MAU** | 月活跃用户数 | 当月打开App的独立用户数 |
| **次日留存率** | 次日回访比例 | 次日活跃用户 / 新增用户 |
| **付费率** | 付费用户占比 | 付费用户数 / 总用户数 |
| **ARPU** | 平均每用户收入 | 总收入 / 总用户数 |
| **ARPPU** | 付费用户平均收入 | 总收入 / 付费用户数 |
| **续费率** | 订阅续费比例 | 续费用户数 / 到期用户数 |
